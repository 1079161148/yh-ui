# 旗舰场景

YH-UI 的差异化价值，不在于单个组件，而在于它能否让团队更快做出完整产品面。

第一阶段应该围绕 3 个旗舰场景来建立认知，每个场景都要像真实产品，而不是散落的 demo 片段。

## 1. AI 工作台

### 目标用户

- AI SaaS
- Copilot 类产品
- 内部智能助手
- 多面板分析工作台

### 应该展示的能力

- `AiChat`、`AiBubble`、`AiConversations`
- `AiSender`、`AiEditorSender`
- `AiThoughtChain`
- `AiCodeBlock`、`AiCodeEditor`、`AiCodeRunner`
- `AiArtifacts`、`AiAttachments`、`AiSources`
- `@yh-ui/request` 的流式请求、队列、重试

### 最小产品面

- 左侧会话列表
- 中间多轮对话区
- 右侧工件 / 来源 / 任务面板
- 顶部模型与工作区切换
- 错误、取消、重试、引用展开等真实状态

### 当前仓库落地状态

这一场景现在已经以 `apps/ai-ops-starter` 中的 `Workspace` 路由开始落地，而不是停留在文档定义层。

当前这一路由已经具备：

- 左侧会话 rail
- 中间生成式主面板
- 右侧思维链与来源侧栏
- prompt 快捷入口
- artifact 抽屉

这意味着后续针对 AI 组件的修复，不再只看单组件 demo，而是可以直接在真实工作台组合里验收。

### 验收重点

- 长对话滚动稳定
- 键盘操作完整
- 代码、附件、引用与文本共存不混乱
- 流式状态和失败恢复足够清楚

## 2. Flow 自动化控制台

### 目标用户

- 自动化平台
- AI Agent 编排平台
- 工作流设计器
- 内部流程工具

### 应该展示的能力

- `@yh-ui/flow`
- 节点、边、控制器、缩略图、背景
- 自动布局、导入导出、截图
- AI 节点配置面板
- 主题系统与多状态侧栏

### 最小产品面

- 中央大画布
- 左侧节点物料区
- 右侧属性 / 配置面板
- 顶部运行、导入、导出、版本状态
- 视口控制、缩放、框选、撤销重做

### 验收重点

- 大画布操作流畅
- 导入导出可恢复状态
- 选择、连线、缩放和拖拽没有明显不稳定点
- 节点配置和画布行为是一个统一产品体验

## 3. 重数据运营后台

### 目标用户

- 电商运营
- 中后台管理
- 数据审核平台
- 复杂权限与多表单协同系统

### 应该展示的能力

- `YhTable`
- 虚拟滚动、列拖拽、列宽调整、固定列
- 筛选、排序、分页、批量操作
- 导入、导出、打印
- `FilterBar`、`Descriptions`、业务卡片

### 最小产品面

- 顶部筛选与工具栏
- 中央表格与统计信息
- 抽屉 / 对话框详情
- 列配置与个性化偏好持久化
- 导出与打印入口

### 验收重点

- 大数据量滚动稳定
- 列配置和筛选状态持久化
- 打印与导出结果可信
- 桌面端和窄视口都有明确策略

## 统一要求

| 维度     | 要求                                                     |
| -------- | -------------------------------------------------------- |
| 产品感   | 先展示完整工作流，不只展示孤立组件                       |
| 可验证性 | 每个场景都要能通过 smoke / a11y / visual regression      |
| 复用性   | 场景里沉淀下来的布局和页面块，要能进入 starter 或 blocks |
| 文档化   | 每个场景都要说明目标用户、依赖模块、验收标准             |

## 第一阶段的工作边界

当前阶段先做：

- 场景定义
- 场景入口页
- 信息架构
- 组件组合方案
- 验收标准

暂时不要求一口气把 3 个场景全部做成完整独立产品仓库，但后续 starter 与 showcase 都要围绕这 3 类场景推进。
